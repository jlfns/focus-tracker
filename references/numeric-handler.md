# A: 数值查询 Handler (numeric-handler)

策略：权威来源查询 → 数值提取 → 趋势判断 → 多源交叉验证。

## Prompt 模板

执行以下 Prompt。替换 `{keyword}`、`{sub_category}`、`{current_date}`（当日真实日期，YYYY-MM-DD 格式）。

**执行前必须先联网搜索**：搜索「{keyword} 最新价格 {current_date}」，至少用 2 组不同关键词交叉验证。

```
你是一个专业的数值查询助手。请根据用户关注的内容，结合联网搜索结果，提供最新、准确的数值数据。

## 当前日期
{current_date}

## 关注项信息
- 关键词：{keyword}
- 子类别：{sub_category}

## 输出要求
请以 JSON 格式输出，严格遵循以下结构：
{
  "value": "核心数值（含单位）",
  "change": "涨跌幅或变化趋势",
  "unit": "数值单位",
  "data_date": "数据的实际日期（YYYY-MM-DD），必须标注",
  "sources": [
    {
      "name": "来源名称",
      "value": "该来源数值",
      "url": "来源链接（如有）"
    }
  ],
  "extra": {
    "high": "最高值（如适用）",
    "low": "最低值（如适用）",
    "freshness": "fresh/stale/unknown — 数据是否接近当前日期",
    "note": "备注说明"
  }
}

## 重要约束
1. 必须对比 data_date 与当前日期 {current_date}。若数据早于 {current_date} 超过 1 天，在 note 中明确标注"数据非实时，最后更新于 X"
2. 无法提供接近 {current_date} 的最新数据时，在 note 中诚实说明"知识截止至 YYYY-MM，无法提供实时数据"
3. 不得编造当前日期之后的未来数据
4. 至少提供 2 个来源进行交叉验证
5. 仅输出 JSON，不要输出其他内容
```

## 纠错验证：多源偏差检测

解析 LLM 返回的 JSON，执行以下验证：

1. **日期新鲜度**：对比 `data_date` 与当前日期。超过 1 天 → 标记 `freshness = "stale"`
2. **多源交叉检查**：对比主 `value` 与 `sources[]` 中的各来源数值
3. **偏差计算**：`|来源值 - 主值| / 主值`
4. **阈值**：5%（0.05）
5. **判定**：
   - 所有来源偏差 <5% → ✅ 理由："多源交叉验证通过"
   - 任一来源偏差 5%-10% → ⚠️ 理由："源间偏差 {x}%，存在轻微分歧"
   - 任一来源偏差 >10% → ❌ 理由："源间偏差 {x}%，数据可能异常"
   - 仅单一来源 → ⚠️ 理由："仅单一来源，无法交叉验证"

## 输出格式

```
📊 {keyword} | {sub_category}
━━━━━━━━━━━━━━━━━━━━━━
数值：{value} {change}
时间：{data_date}
来源：{source1}, {source2}
验证：{✅/⚠️/❌} {reason}
```

❌ 时追加：`⚠️ 数据可能异常，建议核实后再做决策。`
