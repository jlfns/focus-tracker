# B: 信息聚合 Handler (aggregate-handler)

策略：多源采集 → 可信度评分 → 一致性检测 → 结构化摘要。

## Prompt 模板

执行以下 Prompt。替换 `{keyword}`、`{sub_category}`、`{current_date}`（当日真实日期，YYYY-MM-DD 格式）。

**执行前必须先联网搜索**：搜索「{keyword} 最新 {current_date}」，多组关键词获取多样化来源。

```
你是一个专业的信息聚合助手。请根据用户关注的内容，结合联网搜索结果，聚合多来源的最新信息，并对每条信息标注可信度等级。

## 当前日期
{current_date}

## 关注项信息
- 关键词：{keyword}
- 子类别：{sub_category}

## 输出要求
请以 JSON 格式输出，严格遵循以下结构：
{
  "topic": "关注主题",
  "summary": "一句话摘要",
  "data_freshness": "fresh/stale/unknown",
  "items": [
    {
      "title": "信息标题",
      "content": "核心内容摘要（100字内）",
      "source": "来源名称",
      "source_level": "L1/L2/L3/L4",
      "url": "来源链接（如有）",
      "published_at": "发布时间（ISO 8601）"
    }
  ],
  "consistency": "一致/有争议/冲突",
  "trend": "上升/平稳/下降/不确定",
  "extra": {
    "total_sources": "信息来源总数",
    "note": "备注说明"
  }
}

## 重要约束
1. 必须对比每条信息的 published_at 与当前日期 {current_date}。若大部分信息早于 {current_date}，data_freshness 标注为 "stale"
2. 无法提供接近 {current_date} 的最新信息时，在 note 中说明"知识截止至 YYYY-MM，数据可能不是最新"
3. 不得编造当前日期之后的事件
4. 至少聚合 3 条相关信息
5. 每条信息必须标注 source_level
6. 来源尽量多样化
7. 仅输出 JSON
```

## 来源可信度等级

- L1 (1.0)：官方渠道、权威媒体、政府网站 — 例：人民日报、新华社、央视、政府公报
- L2 (0.8)：知名媒体、行业机构、认证账号 — 例：财新、第一财经、36氪、极客公园、GitHub官方
- L3 (0.6)：自媒体、博客、论坛、社区、认证大V — 例：知乎高赞、微博大V、B站UP主
- L4 (0.3)：匿名来源、未验证消息、小道消息、个人社交媒体

## 纠错验证：可信度评分 + 一致性检测

解析 LLM 返回的 JSON，执行以下验证：

1. **日期新鲜度**：检查 `data_freshness` 字段。"stale" → 验证降一级
2. **可信度评分**：每条 source_level 对应权重 L1=1.0, L2=0.8, L3=0.6, L4=0.3，总分 = 所有权重和 / 条数
3. **一致性检查**（LLM 已标注，复核）：
   - "一致"：≥3 个独立来源说法相同
   - "有争议"：多数一致但有分歧
   - "冲突"：来源间矛盾
4. **判定**：
   - 评分 ≥ 0.8 且一致 → ✅ 理由："高可信度，多源一致"
   - 评分 ≥ 0.6 且有争议 → ⚠️ 理由："可信度可接受，但存在分歧"
   - 评分 ≥ 0.6 且一致 → ✅ 理由："可信度可接受，来源一致"
   - 评分 < 0.6 或冲突 → ❌ 理由："可信度不足或信息冲突"

## 输出格式

```
📰 {topic} | {sub_category}
━━━━━━━━━━━━━━━━━━━━━━
可信度：{score}/10 | 一致性：✅/⚠️/❌
摘要：{summary}
━━━━━━━━━━━━━━━━━━━━━━
信息来源：
  [L{n}] {source}: {content_summary}
━━━━━━━━━━━━━━━━━━━━━━
趋势：{trend}
验证：{✅/⚠️/❌} {reason}
```

有争议时追加：
```
⚠️ 分歧点：
  - {分歧1}
  - {分歧2}
```
